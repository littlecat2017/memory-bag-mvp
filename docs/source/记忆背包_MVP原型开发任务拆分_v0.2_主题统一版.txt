一、MVP 原型目标

MVP-1 只验证一个问题：
《记忆背包》的核心循环是否能让玩家把记忆当作关系凭证来处理。

也就是：
玩家看着勇者自动前进和战斗时，是否愿意认真处理“这段记忆连接着谁、保留或丢掉会让世界如何回应”的选择。

MVP-1 不追求：
1. 完整故事。
2. 正式美术。
3. 完整五结局。
4. 合成系统。
5. 商业化包装。

MVP-1 必须在 10-15 分钟内让玩家经历：
1. 初始记忆选择。
2. 自动前进。
3. 自动战斗。
4. 背包满时取舍。
5. 至少一次“数值收益 vs 关系凭证”的冲突。
6. Boss 前是否献出名字。
7. 根据名字和出发理由状态变化的短结尾。

MVP-1 的关键不是让玩家觉得失去记忆很沉重。
关键是让玩家意识到：
丢掉一段记忆后，世界不再用同一种方式回应勇者。


二、MVP-1 范围

包含内容：
1. 序章：故乡。
2. 第一章：遗忘森林，1000m。
3. 自动行进系统。
4. 自动战斗系统。
5. 4 格记忆背包。
6. 记忆获得、放弃、丢弃、替换。
7. 核心记忆二次确认。
8. JSONL 脚本播放。
9. choice.options[].effects 执行。
10. 3 场普通战斗。
11. 1 场 Boss：忘名猎人。
12. 最小世界反馈：
    - 名字栏是否存在。
    - 关键 NPC 或旁白是否还能以关系回应勇者。
    - 结尾是否有人能叫出勇者的名字。

使用记忆：

初始候选：
- mem_mothers_soup：母亲做的热汤
- mem_wooden_sword：第一次拿起木剑
- mem_reason_to_depart：出发的理由
- mem_my_name：我的名字

第一章新增：
- mem_someone_waits：有人等我
- mem_abandoned_afternoon：被抛下的下午
- mem_no_more_explaining：不再解释
- mem_empty_nameplate：空名牌

使用敌人：
- enemy_hollow_wolves
- enemy_nameless_deer
- enemy_hollow_warden
- boss_nameless_hunter

暂不包含：
1. 记忆合成。
2. 第二章、第三章、终章。
3. 完整五结局。
4. 离线收益。
5. 复杂商店。
6. 成就。
7. 装备系统。
8. 技能树。
9. 正式音频系统。
10. 正式美术资源。


三、推荐技术路线

推荐 Unity 或 Godot。

本拆分不强绑定引擎。
核心要求是数据驱动、快速验证、方便调试。

Unity 建议结构：

Assets/
  Scripts/
    Core/
    Data/
    Runtime/
    UI/
    Debug/
  Data/
    json/
    tables/
  Art/
    Prototype/
  Scenes/
    MVP.unity

Godot 建议结构：

res://
  scripts/
    core/
    data/
    runtime/
    ui/
    debug/
  data/
    json/
    tables/
  art/
    prototype/
  scenes/
    mvp.tscn

MVP 技术原则：
1. 数据驱动优先。
2. UI 简单但必须清楚。
3. 每个关键系统都能在调试面板直接触发。
4. 不做复杂架构。
5. 不做正式资源管线。
6. 记忆、敌人、章节流必须表驱动。
7. 原型阶段允许少量流程硬编码，但不能硬编码记忆意义和结局条件。


四、数据文件拆分

以《记忆背包_MVP数据表_v0.2_主题校准版.txt》为准，拆成 6 类数据。

1. script_extract_mvp.json

来源：
记忆背包_完整脚本_JSONL_v1.3_完整主题统一版.txt

用途：
定义 MVP 截取范围。

MVP 截取：
- meta
- event_schema
- condition_syntax
- asset_manifest
- character_def
- tutorial
- memory_def 中的 8 张 MVP 记忆
- prologue 事件 P0001-P0037
- forest 事件 F0001-F0040

2. script_events_mvp.jsonl

用途：
剧情播放、选择、战斗触发、记忆事件。

要求：
每行一个 JSON 对象。
启动时校验 id 唯一、choice target 存在、memory 引用存在。

3. memories.json

必须字段：

name

effect_text

discard_text
stats
relation_target
relation_type
obligation
owned_world_response

ui_loss_hint

4. enemies.json

必须字段：
id
name
max_hp
atk
def
attack_interval
exp_reward
gold_reward
tags
mechanics

5. chapter_flow.json

必须字段：
chapter_id
name
distance
nodes

node 字段：
progress
type
event_id

6. balance.json

用途：
玩家初始数值、升级曲线、背包容量、核心记忆确认文案、名字栏表现。

7. mvp_endings.json

用途：
MVP 结尾分支。

至少包含：
- name_kept_reason_kept
- name_lost_reason_kept
- name_kept_reason_lost
- name_lost_reason_lost

六、核心系统任务

Milestone 0：空工程与数据校验

目标：
能启动游戏，读取数据，输出 validation errors: 0。

任务：
1. 创建工程。
2. 创建 MVP 主场景。
3. 导入 JSON/JSONL 数据。
4. 实现 DataRegistry。
5. 校验 memory id 唯一。
6. 校验 enemy id 唯一。
7. 校验 event id 唯一。
8. 校验 choice target 存在。
9. 校验 effects 引用的 memory id 存在。
10. 校验每张 MVP 记忆都有 relation_target、relation_type、obligation、owned_world_response、discard_world_response、ui_loss_hint。

验收：
启动后日志显示数据加载成功，并列出记忆、敌人、事件数量。


Milestone 1：脚本播放器

目标：
能播放序章和森林脚本。

任务：
1. 实现 line 事件播放。
2. 实现 tutorial 事件播放。
3. 实现 choice 事件播放。
4. 实现条件显示。
5. 实现 effects 执行。
6. 实现 memory_offer、memory_gain、memory_discard。
7. 支持 bg、speaker、portrait 字段。
8. 支持无正式资源时的占位显示。

验收：
可以从 P0001 播放到 P0037。
选择开局路线后，背包状态正确。


Milestone 2：记忆背包

目标：
背包取舍成为 MVP 的核心交互。

任务：
1. 实现 4 格背包 UI。
2. 显示记忆名称、标签、数值效果。
3. 显示关系对象和承诺摘要。
4. 显示丢弃提示。
5. 背包满时打开替换界面。
6. 支持“放弃新记忆”。
7. 支持核心记忆二次确认。
8. 丢弃核心记忆后更新 UI 状态。
9. 丢弃“我的名字”后擦除名字栏。

验收：
F0010 获取“有人等我”时，如果背包已满，能正确进入替换流程。
丢弃不同记忆后，状态、UI、日志都能区分。


Milestone 3：自动行进

目标：
玩家能看到勇者自动推进到事件节点。

任务：
1. 实现距离条。
2. 实现自动前进速度。
3. 实现按 progress 触发节点。
4. 战斗时暂停或减缓前进。
5. 事件播放时暂停前进。
6. 支持调试跳转到指定 progress。

验收：
能从森林 0m 自动推进到 1000m，并按 chapter_flow 触发事件。


Milestone 4：自动战斗

目标：
战斗足够简单，但能体现记忆差异。

任务：
1. 实现玩家和敌人基础数值。
2. 实现自动攻击。
3. 实现伤害公式。
4. 实现胜利奖励。
5. 实现记忆 stats 加成。
6. 实现对无名敌人的特殊加成。
7. 实现战斗日志。
8. 让部分战斗日志引用记忆关系。

验收：
带“第一次拿起木剑”时攻击更高。
带“有人等我”时持续恢复。
献出名字获得“空名牌”后，对忘名猎人更有优势。


Milestone 5：世界反馈

目标：
让记忆丢弃不只是数值变化。

任务：
1. 实现名字栏。
2. 实现关系反馈日志。
3. 实现丢弃后场景文本变化。
4. 实现条件台词。
5. 实现 Boss 前选择条件。
6. 实现结尾文本分支。

验收：
丢弃“母亲做的热汤”后，营火仍能作为功能存在，但文本不再指向母亲。
丢弃“我的名字”后，Boss 和结尾不能再以名字回应勇者。


Milestone 6：Boss 与 MVP 结尾

目标：
完成第一章闭环。

任务：
1. 实现忘名猎人战前事件。
2. 实现“拒绝献名”“献出名字”“沉默”选择。
3. 实现 mem_empty_nameplate 获得。
4. 实现 Boss 战。
5. 实现 Boss 后结尾判定。
6. 显示 mvp_endings.json 中对应文本。

验收：
至少能跑出四种结尾方向：
1. 名字保留，理由保留。
2. 名字丢失，理由保留。
3. 名字保留，理由丢失。
4. 名字丢失，理由丢失。


Milestone 7：存档与调试

目标：
提高迭代效率。

任务：
1. 实现本地存档。
2. 保存背包、进度、已触发事件、关键选择。
3. 加载后恢复状态。
4. 实现调试面板。
5. 调试面板可添加/移除记忆。
6. 调试面板可跳到 P0034、F0010、F0021、F0034、F0040。
7. 调试面板可强制进入指定结尾。

验收：
退出重进后，背包和章节进度正确。
开发者能 30 秒内跳到 Boss 前测试选择。


七、UI 需求

主界面：
1. 勇者状态。
2. 距离条。
3. 当前章节。
4. 自动战斗区域。
5. 战斗日志。
6. 记忆背包。
7. 当前名字栏。

记忆卡显示：
1. 名称。
2. 标签。
3. 数值效果。
4. 关系对象。
5. 承诺摘要。
6. 丢弃后果提示。

替换界面：
1. 新记忆详情。
2. 当前背包 4 张卡。
3. 每张卡的丢弃提示。
4. 放弃新记忆按钮。
5. 核心记忆二次确认弹窗。

禁止：
不要只显示攻击、防御、恢复。
如果玩家看不到关系对象和承诺，这个 UI 就没有完成。


八、测试路径

标准路径：
1. 新游戏。
2. 标准开局。
3. 到 F0010，拾取“有人等我”。
4. 背包满，丢弃“母亲做的热汤”。
5. 确认营火或回响文本变化。
6. 到 F0021，拾取“不再解释”。
7. 背包满，丢弃“第一次拿起木剑”。
8. Boss 前拒绝献名。
9. 打败 Boss。
10. 看到名字保留结尾。

献名路径：
1. 标准开局。
2. Boss 前选择献出名字。
3. mem_my_name 进入 discarded。
4. mem_empty_nameplate 进入 owned。
5. 名字栏被擦除。
6. Boss 战对无名敌人伤害提高。
7. 结尾进入无名方向。

轻装路径：
1. P0034 选择轻装开局。
2. mem_mothers_soup 进入 discarded。
3. 背包只有 3 张初始记忆。
4. F0010 拾取新记忆时不打开替换界面，直接入包。

关系反馈路径：
1. 丢弃“出发的理由”。
2. 确认终章或 MVP 结尾中无法完整回应莉娅托付。
3. 保留“出发的理由”再跑一次。
4. 对比结尾文本。


九、完成定义

MVP-1 完成必须同时满足：
1. 可以从新游戏开始，无报错跑完序章和第一章。
2. 玩家至少遇到两次记忆取舍。
3. 背包替换和核心记忆二次确认可用。
4. 自动战斗能根据记忆组合产生可感知差异。
5. 忘名猎人 Boss 前献名选择可用。
6. Boss 后至少有四种名字/理由组合结尾。
7. 丢弃关键记忆后 UI 或文本有可见反馈。
8. 有调试面板可以快速跳到关键节点。
9. 一轮完整流程控制在 10-15 分钟。
10. 至少找 3 个外部玩家或朋友试玩。

试玩后必须记录：
1. 玩家在哪个选择停顿。
2. 玩家丢了哪张记忆。
3. 玩家是否能说出这张记忆连接着谁。
4. 玩家是否注意到世界反馈变化。
5. 玩家是否愿意再玩一遍。

十一、风险点和处理

风险一：玩家只看数值。

处理：
记忆卡必须显示关系对象和承诺。
丢弃后必须有世界反馈。
Boss 和结尾必须回应关键记忆状态。

风险二：玩家觉得文本和玩法分离。

处理：
关系字段进入 UI、战斗日志、事件条件和结尾判定。
不要让主题只存在于旁白里。

风险三：战斗太抢戏。

处理：
降低战斗操作复杂度。
让战斗成为检验记忆组合的压力，而不是独立玩法中心。

风险四：战斗太无聊。

处理：
缩短等待。
提高事件密度。
让记忆触发在战斗日志中出现。

风险五：玩家无脑丢核心记忆。

处理：
核心记忆二次确认。
丢弃后 UI 明显变化。
Boss 和结尾强烈反馈。
不要简单给核心记忆加攻击力。

风险六：系统过早膨胀。

处理：
MVP-1 不做合成、不做终章、不做正式美术。
第一章不好玩，不扩第二章。


十二、MVP-1 后的决策

如果试玩结果显示：
玩家会停顿、会讨论记忆连接着谁、愿意再玩。

进入 MVP-2：
开发第二章、合成系统、灰骑士。

如果试玩结果显示：
玩家只看数值，不在乎关系反馈。

暂不做第二章。
重做记忆卡表达、丢弃反馈、Boss 回应。

如果试玩结果显示：
玩家喜欢故事，但觉得战斗等待无聊。

缩短行进时间。
增加事件密度。
让战斗更快结算。

如果试玩结果显示：
玩家不理解系统。

重做 UI 和前 5 分钟教学。
不要继续加内容。
